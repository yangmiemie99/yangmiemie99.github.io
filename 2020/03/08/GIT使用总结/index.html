GIT使用总结

发表于 2020-03-08 | 分类于 版本控制系统

GIT介绍

Git是一个快速的版本分布式系统。

GIT对象模型

SHA

所有用来表示项目历史信息的文件，是通过一个40个字符的“对象名”来索引的，看起来像这样：
6ff87c4664981e4397625791c8ea3bbb5f2279a3。

每个对象名都是对“对象”内容做SHAI(SHA1
是一种密码学的哈希算法)计算的来的。所以不同内容的对象不可能有相同的对象名。

  • Git只要比较对象名,就可以很快的判断两个对象是否相同。
  • 因为在每个仓库(repository)的“对象名”的计算方法都完全一样,如果同样的内容存在两个不同的仓库
    中,就会存在相同的“对象名”下。
  • Git还可以通过检查对象内容的SHA1的哈希值和“对象名”是否相同,来判断对象内容是否正确。

对象

每个对象(object) 包括三个部分:类型,大小和内容。大小就是指内容的大小,内容取决于对象的类型,有四种类
型的对象:”blob”、”tree”、 “commit” 和”tag”。

  • blob”用来存储文件数据,通常是一个文件。

    • 可以使用git show命令来查看一个blob对象里的内容。e.g. git show 6ff87c4664981..
    • 一个blog对象就是一块二进制数据，没有任何其他属性。

  • “tree”有点像一个目录,它管理一些“tree”或是 “blob”(就像文件和子目录)

    • 可以使用git show命令查看对象，使用 git ls-tree能看到更过的细节。
    • 一个tree对象有一串指向blob对象或者其他tree的指针，它一般用来表示内容之间的层次关系。
    • 一个tree对象包括一串(list)条目,每一个条目包括:mode、对象类型、SHA1值 和名字(这串条目是按名字排序的)。

  • 一个“commit”对象只指向一个”tree”对象,并带有描述信息。

    • 可以用 –pretty=raw 参数来配合 git show 或 git log 去查看某个提交(commit)
    • 组成部分
      • 一个 tree 对象: tree对象的SHA1签名, 代表着目录在某一时间点的内容.
      • 父对象 (parent(s)): 提交(commit)的SHA1签名代表着当前提交前一步的项目历史.如果没有副对象，叫做“根提交”。
      • 作者：修改人的名字，还有修改日期
      • 提交人：实际创建提交的人的名字，同时带有提交日期。
      • 注释 用来描述这次提交
    • 一般用 git commit 来创建一个提交(commit), 这个提交(commit)的父对象一般是当前分支(current HEAD), 同时把存
      储在当前索引(index)的内容全部提交.

  • 一个“tag”是来标记某一个提交(commit) 的方法。

    • 可以用git cay-file命令查看这些信息
    • 包含对象名，对象类型，标签名，标签创建人的名字，还有可能包括签名的消息。

  • 用伪代码表示对象，哈哈，清楚多了

    • type object = blob | tree |commits

    • objects = map<string.object>

    • type blob = array<byte>

    • type tree = map<string,tree|blob>

与 SVN的区别

Git与你熟悉的大部分版本控制系统的差别是很大的。也许你熟悉Subversion、CVS、Perforce、Mercurial 等等,他们使用 “ 增量文件系统 ” (Delta Storage systems), 就是说它们存储每次提交(commit)之间的差异。Git正好与之相反,它会把你的每次提交的文件的全部内容(snapshot)都会记录下来。

GIT目录与工作目录

GIT目录

‘Git目录’是为你的项目存储所有历史和元信息的目录 - 包括所有的对象(commits,trees,blobs,tags), 这些对象指向不同的分支。

工作目录

Git的工作目录存储着你现在签出用来编辑的文件，当你在项目的不同分支间切换时，工作目录里的文件经常会替换和删除。说有的历史信息都保存在Git目录中;工作目录只用来保存临时签出文件

GIT索引

Git索引是一个你的工作目录和项目仓库的暂存区。有了它，可以把很多内容的修改一起提交，那么提交的是当前索引（index）里的内容，而不是工作目录中的内容。

查看索引

git status命令查看索引中的内容，可以看到那些文件被暂存了（就是在你的git索引中），哪些被修改了但是没有被暂存，还有那些文件没有被跟踪。

如果完全掌握了索引(index), 你就一般不会丢失任何信息, 只要你记得名字描述信息(name of the tree that it
described)就能把它们找回来.

正常的工作流程

  • 修改文件将他们更新的内容添加到索引中。

    • git add file1 file2 file3
    • git add -p 以互动的方式让我确认需要提交的更改

  • 现在为commit做好了准备，你可以使用git diff命令加上–cached参数，看看那些文件将被提交

    • git diff -cached

    • (如果没有–cached参数,git diff 会显示当前你所有已做的但没有加入到索引里的修改.)

    • 也可以使用git status命令来获取当前目录的一个情况

  • 隐藏修改

    • git stash 隐藏当前修改，将工作哦目录状态还原为上次的提交

  • git stash pop 撤销上次的隐藏

  • 提交

    • git commit

    • 这会提示你输入本次修改的注释,完成后就会记录一个新的项目版本.

  • 忽略一些垃圾文件，e.g. 编译产生的.o等二进制文件，不add到索引内

    • vim .gitignore

  • 或者git commit -a

  • 这会自动把所有内容修改的文件（不包括新创建的文件）都添加到索引中，并且把他们同时提交。

  • commit注释技巧

    • 最好以一行短句子作为开头，来描述这次commit所作出的修改;然后空一行把详细的注释写清楚。这样哪个就可以很方便的用工具把commit注释变为email通知，第一行作为标题，剩下的作为正文

Git跟踪的是内容不是文件

add指令：告诉系统开始跟踪一个文件的改动。git add不但是用来添加不在版本控制器中的新文件，也用于添加已在版本控制器中但是刚修改过的文件;在这两种情况下，Git都会当前文件的快照并把内容暂存在索引中，为下一次comiit做好准备。

git checkout experimental 对分支进行修改不改动其他的分支。

合并

git merge experimental

如果这个两个分支间的修改没有冲突(conflict), 那么合并就完成了。

合并冲突的解决

冲突的来源

冲突：同一个文件在远程分支和本地分支按照不同的方式被修改了，输出以下内容

在有问题的文件上会有冲突标记，在你手动解决完冲突之后就可以把文件添加到索引中去,然后提交。

执行git status 会显示这些文件没有被合并，这些有冲突的文件里面添加冲突标识符

如果有冲突，输入以下命令查看房前有那些文件冲突。

git diff

解决冲突

编辑解决冲突，删除冲突标识符

编辑解决文件冲突就可以提交了

git merge continue

git commit -a

查看项目历史

提交合并之后可查看gitk

执行gitk之后会有一个很漂亮的图形界面显示项目的历史。

删除分支

合并之后就可以删除我的experimental分支了

git branch -d experimental

  • 强制删除莫个分支就用
    • git branch -D

撤销一个合并

如果合并的状态不好，向放弃当前的修改,用一下命令合并之前的状态

git reset --hard HEAD

已经把合并后的代码提交，还是想把他们撤销

git reset --hard ORIG_HEAD

这条命令某些情况很危险，如果你把一个已经被另一个分支合并的分支删了。那么在以后合并相关分支会出现错误

快速向前合并

通常,一个合并会产生一个合并提交(commit), 把两个父分支里的每一行内容都合并进来。
但是,如果当前的分支和另一个分支没有内容上的差异,就是说当前分支的每一个提交(commit)都已经存在另一个分支里了,git 就会执行一个“快速向前”(fast forward)操作;git 不创建任何新的提交(commit),只是将当前分支指向合并进来的分支。

日志排序

你也可以把日志记录按一些不同的顺序来显示。

git 日志从最近的提交开始，并且从这里开始向他们父分支回溯。然而git历史可能包括多个互不相关联的开发线路，指定特殊顺序，可以为git log 命令添加顺序参数。

  • 按默认情况,提交(commits)会按逆时间(reverse chronological)顺序显示。
  • –topo-order按照拓扑顺序显示git仓库的提交日志，“开发线”都会集合在一起。
  • –data-order 按照日志的提交顺序来排序

比较提交

比较两个版本的差异

找出‘master’,‘test’的共有 父分支和’test’分支之间的差异

git diff 是一个难以置信的有用的工具,可以找出你项目上任意两点间 的改动,或是用来查看别人提交进来的新分
支。

查看哪些内容会被提交

git diff

上述命令会显示在当前的工作目录内的，没有staged（添加到索引中）

git diff --cached

上述命令会显示当前索引和上次提交间的差异，这些内容在不带“-a”参数运行git commit命令时就会被控制提交。

上述命令会显示你的工作目录与上次提交时间的所有差别，这跳命令所显示内容都会被git commit -a所提交。

更多比较选项

当前目录与另一个分支的差别

git diff HEAD -- ./lib

不看详细差别，统计那些文件被改动

git diff stat

GIT标签

轻量级标签

我们可以用 git tag不带任何参数创建一个标签(tag)指定某个提交(commit):

git tag stable-l 1b2e3be3nn

这样我们可以用stable-l作为提交“1b2e3be3nn”的代称

前面这样创建的是一个“轻量集标签”，这种分支通常是不移动的。

标签对象

如果有 “-a”, “-s” 或是 “-u “ 中间的一个命令参数被指定,那么就会创建 一个标签对象,并且需要一个标签消息(tag
message). 如果没有”-m “ 或是 “-F “ 这些参数,那么就会启动一个编辑器来让用户输入标签消息(tag message).

git tag -a stable-1 1b2e1d63ff
标签对象可以指向任何对象,但是在通常情况下是一个提交(commit).
