文件系统和文件

发表于 2020-03-28 | 分类于 操作系统

#

文件系统和文件

文件系统：一种用于持久性存储的系统抽象

  • 在存储器上：组织、控制带行访问和检索数据
  • 大多数计算机系统包含文件系统

文件：文件系统中一个单元的相关数据在操作系统中的抽象

文件系统的功能

分配文件磁盘空间

  • 管理文件块（哪块属于哪一个文件）
  • 管理空闲块 （哪一块是空闲的）
  • 分配算法

管理文件集合

  • 定位文件及内容
  • 命名：通过名字找到文件的接口
  • 最常见：分层文件系统
  • 文件系统类型（组织文件的不同方式）

提供的便利及特性

  • 保护：分层来保护数据安全
  • 可靠性/持久性：保持文件的持久即时发生崩溃，媒体错误，攻击等。

文件属性

名称，类型，位置，大小，保护，创建者，创建时间，最近修改时间…

文件头

  • 在存储元数据中保存了每个文件的信息
  • 保存文件属性
  • 跟踪哪一块存储属于逻辑上文件结构的哪个偏移

文件描述符

内核跟踪每个进程打开的文件

  • 操作系统为每个进程维护一个打开文件表
  • 一个打开文件描述符是这个表中的索引

需要元数据来管理打开文件

  • 文件指针：指向最近的一次读写位置
  • 文件打开次数：所有进程都关闭了文件，润序将其从打开文件表中删除
  • 文件的磁盘位置：缓存数据访问信息
  • 访问权限：每个程序访问模式信息

怎么实现？

高层操作系统不用关心具体存在什么地方，只是对buffer的一个读写，最后由文件系统将堆在内存中的buffer 对应到磁盘中去，完成对磁盘的读写,最后体现在用户对磁盘的持久访问。

在操作系统内部：

怎么将文件和 磁盘块 数据对应起来，一般扇区512字节，内存中页4k，怎么对应起来？在用户角度不用管这件事，操作系统看到的是底层的一个一个磁盘块，用户看到的是一个抽象的一维线性的数据空间。

例子

  • 当用户说给我2-12字节空间时发生什么？
    • 获取字节所在的磁盘块
    • 返回块内对应部分
  • 当空户要写2-12字节呢？
    • 获取相应块
    • 修改块对对应部分
    • 写回块

用户怎么访问文件

在系统层面需要直到用户的访问模式

  • 顺序访问：按字节依次读取
    • 几乎所有的访问都是这种方式
  • 随机访问：从中间读写
    • 不常用，但是仍然重要，例如，虚拟
    • 内存支持文件：内存页存储在文件中
    • 加速：不希望获取其他无用的字节
  • 基于内容的访问：通过特征
    • 例如数据库是建立在索引内容的磁盘访问上

目录

文件以目录的方式组织起来

目录是一类特殊的文件

  • 每个目录包含了一张表< name ,pointer to file header>
  • 目录和文件是一种树形结构

典型操作

  • 搜索文件
  • 创建文加
  • 删除文件
  • 枚举目录
  • 重命名文件
  • 在文件系统中遍历一个路径

操作系统中只允许内核模式修改目录

  • 确保映射完整性
  • 应用程序能够读目录

挂载

概念：根文件系统之外的其他文件要想能够被访问，都必须通过”关联“置根文件系统上的某个目录来实现，这个关联此操作就是挂载，此目录即为挂载点，解除此关联关系的过程称之为”挂载“。

一个文件系统需要先挂载才能被访问

一个没有挂载的文件系统被挂载在挂载点上。

挂载点：用户看来就是一个目录，代表文件系统根的起始位置，访问文件存储的入口。

文件别名

  • 硬链接：多个文件项指向一个文件
  • 软连接：以快捷方式指向其他文件

虚拟文件系统

分层结构

  • 上层：虚拟（逻辑）文件系统
  • 底层：特定文件系统模块

目的

  • 对所有不同文件系统的抽象

功能

  • 提供相同的文件和文件系统接口
  • 管理所有文件和文件系统关联的数据结构
  • 高校查询例程，遍历文件系统
  • 与特定文件系统模块的交互

完全是内存中的概念，不存在硬盘上。当操作系统创建后，会把虚拟文件系统创建好。

基本数据结构

  • 卷控制块（UNIX:superblock）
    • 每个文件系统一个
    • 文件系统详细信息
    • 块、块大小、空余块、计数/指针等
  • 文件控制块（inode）
    • 每个文件一个
    • 文件详细信息
    • 许可，拥有者，大小，数据位置等
  • 目录节点（dentry）
    • 每个目录一个
    • 将目录项数据结构及属性布局编码成数据结构
    • 指向文件控制块，父节点，项目录列表等

当需要时加载进内存

  • 卷控制块：：当文件系统挂载时进入内存
  • 文件控制块：当文件被访问时进入内存
  • 目录节点：在遍历一个文件路径时进入内存

数据缓存

数据块按需读入内存

  • 提供read()操作
  • 预读：预读取后面的数据块

数据块使用后被缓存

  • 假设数据将会再次被使用
  • 写操作可能被缓存和延迟写入

两种数据块缓存方式

  • 普通缓冲区缓存
  • 页缓存：统一缓存数据块和内存页

分页要求

  • 当需要一个页时才将其载入内存

支持存储

  • 一个页（在虚拟地址空间中）可以被映射到一个本地文件中（在耳机存储中）

文件数据块的页缓存

  • 在虚拟内存中的文件数据被映射成页
  • 文件的读/写操作被转换成对内存的访问
  • 可能导致缺页/设置为脏页
  • 问题：页置换-从进程或文件页缓存中？

打开文件

找到文件，把文件控制块中内容读到内存，把文件控制块的关键信息放到打开文件表中，文件表中设置一个项，将这个项的索引返回给我们的应用程序，基于index进行读写就可以了。文件读写时inode信息会指出文件到底在什么地方，读写时会有一个offset 指出这个文件哪个位置起哪个数据，offset经过文件控制块里的一些转换，将offset转化成磁盘扇区的一个编号。

文件分配

  • 连续分配

    • 文件头指定起始块和长度
    • 优点：
      • 文件读取表现好
      • 高效的顺序和随机访问
    • 劣势：
      • 碎片
      • 文件增长，不得不移动后面的文件

  • 链式分配

    • 文件数据块链表方式存储
    • 文件头包含了第一块和最后一块指针
    • 优点
      • 创建，增大，缩小很容易
      • 没有碎片
    • 缺点
      • 不可能真正的随机访问
      • 可靠性不好，比如断了一个链。。

  • 索引分配

    • 为每个文件创建一个名为索引数据块的非数据数据块
      • 到文件数据块的指针列表
    • 文件头包含了索引数据块
    • 优点
      • 创建，增大，缩小很容易
      • 没有碎片
      • 支持直接访问
    • 缺点
      • 当文件很小时，存储索引的开销
      • 如何处理大文件？索引块有容量限制，多级索引

空闲空间列表

  • 用位图代表空闲数据块列表
    • 1111111110001111101010001111…..
    • 如果i = 0代表数据块是空闲的
  • 使用简单但可能回事一个big vector
    • 160GBdisk -> 40Mblock -> 5Mb bits
    • 然而，如果空闲空间在磁盘中均匀分布，那么找到”0“之前需要扫描n/r
      • n = 磁盘上数据块总数
      • r = 空闲数据块总数
  • 出现问题，分配磁盘块，相应位置为1，还没写回磁盘，掉电。访问出现不一致性问题
    • 解决：先把将磁盘中的bit设置为1,再去分配磁盘块，在将内存中的bit设这为1。

多磁盘管理RAID

  • 分区：硬件磁盘的一种适合操作系统指定格式的划分
  • 卷：一个拥有一个文件系统实例的可访问的存储空间。

问题：单个磁盘访问效率低，可靠性低

  • 使用多个并行磁盘来增加
    • 吞吐量
    • 可靠性和可用性
  • RAID - 冗余磁盘阵列
    • 各种磁盘管理技术
    • RAID levels:不同RAID分类（RAID-0,RAID-1,RAID-5）
  • 实现
    • 在操作系统内核：存储/卷管理
    • RAID硬件控制器（i/O）
  • RAID0，数据分成多个子块，独立存储，提升访问效率

  • RAID1，向两个磁盘写相同数据，从任何磁盘读数据。
  • RAID4，既能提高性能，又能提高速度。数据块级磁盘配有专用一个奇偶校验磁盘。允许从任意一个故障磁盘中恢复。问题：奇偶校验磁盘频繁写操作。
  • RAID5，将奇偶校验块均匀分布在每个磁盘里。
